# 开放世界核心指标

| 指标名                  | 代码中的键 | 含义                                                         | 作用                                     | 好坏方向 | 快速判断方法                                                 |
| ----------------------- | ---------- | ------------------------------------------------------------ | ---------------------------------------- | -------- | ------------------------------------------------------------ |
| Wilderness Impact       | `WI`       | 在开放世界下，未知目标对已知类检测带来的额外干扰强度         | 反映 unknown 是否把 known 检测“搅乱”了   | 越低越好 | 当 `U_R50` 上升时，若 `WI` 也下降或基本不涨，说明 unknown 检出更干净；若 `WI` 大涨，往往是把大量未知误报成已知 |
| Absolute Open-Set Error | `AOSA`     | 代码里返回的是 `total_ose[50]`，实际语义就是日志里的 **A-OSE**：未知目标被错分成已知类的绝对数量 | 直接衡量“unknown→known”误分错误          | 越低越好 | 这是最敏感的开放集错误指标之一；如果它很大，说明虽然看到了东西，但经常把未知认成熟知 |
| Current AP50            | `CK_AP50`  | 当前任务**新引入类别**的平均 AP50                            | 看本阶段新类别学得怎么样                 | 越高越好 | 第一个看“新任务有没有学会”的指标                             |
| Known AP50              | `K_AP50`   | 当前阶段为止所有已知类的平均 AP50                            | 看整体已知类检测能力，以及是否灾难性遗忘 | 越高越好 | 论文主表里一般最该看这个；若它明显掉，说明方法伤害了已知类   |
| Unknown AP50            | `U_AP50`   | unknown 类的 AP50                                            | 辅助衡量 unknown 检测质量                | 越高越好 | 可看，但一般不如 `U_R50 + WI + AOSA` 组合稳定                |
| Unknown Recall50        | `U_R50`    | unknown 类在 IoU=0.5 下的召回率                              | 最直接衡量“未知目标能不能被发现”         | 越高越好 | OWOD 里最关键的 unknown 指标之一；如果一直接近 0，说明 unknown 分支/伪监督基本没起作用 |



# 分组 Precision / Recall 指标

| 指标名                     | 代码中的键 | 含义                                                      | 作用                                  | 好坏方向 | 快速判断方法                                                 |
| -------------------------- | ---------- | --------------------------------------------------------- | ------------------------------------- | -------- | ------------------------------------------------------------ |
| Current Precision50        | `CK_P50`   | 当前任务新类在 IoU=0.5 下的 precision                     | 看当前新类误检多不多                  | 越高越好 | 若 `CK_R50` 高但 `CK_P50` 很低，说明靠大量误检换召回         |
| Current Recall50           | `CK_R50`   | 当前任务新类在 IoU=0.5 下的 recall                        | 看当前新类漏检多不多                  | 越高越好 | 配合 `CK_AP50` 看：召回高但 AP 低，通常是 precision 太差     |
| Known Precision50          | `K_P50`    | 所有已知类的 precision                                    | 看已知类整体误检                      | 越高越好 | `K_AP50` 下降时先看它是否同步下降                            |
| Known Recall50             | `K_R50`    | 所有已知类的 recall                                       | 看已知类整体漏检                      | 越高越好 | `K_AP50` 下降但 `K_P50` 正常时，往往是 recall 掉了           |
| Unknown Precision50        | `U_P50`    | unknown 的 precision                                      | 看 unknown 预测里有多少是真的 unknown | 越高越好 | 若 `U_R50` 高但 `U_P50` 很低，说明 unknown 分支很“躁”        |
| Previous Known AP50        | `PK_AP50`  | 之前阶段旧类的平均 AP50，仅在 `prev_intro_cls > 0` 时返回 | 直接看增量遗忘                        | 越高越好 | T2/T3/T4 很重要；如果它掉得比 `K_AP50` 明显，说明新任务压坏了旧知识 |
| Previous Known Precision50 | `PK_P50`   | 旧类 precision                                            | 分析旧类误检                          | 越高越好 | 和 `PK_R50` 一起看遗忘模式                                   |
| Previous Known Recall50    | `PK_R50`   | 旧类 recall                                               | 分析旧类漏检                          | 越高越好 | 常用来判断是“忘了”还是“保守了”                               |

# 训练与优化指标

| 指标名       | TensorBoard 路径/来源 | 含义                              | 作用                         | 好坏方向                    | 快速判断方法                                      |
| ------------ | --------------------- | --------------------------------- | ---------------------------- | --------------------------- | ------------------------------------------------- |
| 总损失       | `train/total_loss`    | 当前 step 的总训练损失            | 最粗粒度看是否在收敛         | 一般下降更好                | 前几个 epoch 明显下降通常正常；长期剧烈震荡要警惕 |
| 各项加权损失 | `train_scaled/*`      | 乘上 loss weight 后参与优化的损失 | 看“真正影响反向传播”的损失项 | 一般下降更好                | 哪项长期最大，哪项就是主矛盾                      |
| 各项原始损失 | `train_unscaled/*`    | 未乘权重的原始损失                | 看损失本身尺度是否异常       | 视项而定                    | 用来判断是不是 weight 配置掩盖了真实问题          |
| 分类误差     | `class_error`         | 已知类分类错误率                  | 看已知类分类状态             | 越低越好                    | 如果 box loss 正常但它很高，多半是分类没学好      |
| 学习率       | `lr`                  | 当前学习率                        | 监控训练调度                 | 无绝对好坏                  | 看是否按预期在 `lr_drop` 附近变化                 |
| 梯度范数     | `grad_norm`           | 当前梯度总范数                    | 监控训练稳定性               | 过大不好，长期接近 0 也不好 | 突然飙升常见于损失爆炸；长期极小可能学不动        |

# 方法使用的OWOD训练损失

| 指标名                      | 常见键名                       | 含义                                     | 作用                      | 好坏方向 | 快速判断方法                                                 |
| --------------------------- | ------------------------------ | ---------------------------------------- | ------------------------- | -------- | ------------------------------------------------------------ |
| 分类损失                    | `loss_ce`                      | 已知类分类损失                           | 保证 known 分类能力       | 越低越好 | 若它降不下去，已知类 AP 往往起不来                           |
| 框回归损失                  | `loss_bbox`                    | L1 框损失                                | 学定位                    | 越低越好 | 与 `loss_giou` 一起看；两者都不降说明定位没学好              |
| GIoU 损失                   | `loss_giou`                    | 几何重叠损失                             | 强化定位质量              | 越低越好 | 和 bbox 一起看更有意义                                       |
| 概率物体性损失              | `loss_obj_ll`                  | PROB 的 objectness likelihood 损失       | 学“是不是物体”            | 越低越好 | 这是 PROB/UOD 主线损失之一，没降下来通常 unknown 也难做      |
| matched-known 负未知性      | `loss_unk` 或 `loss_unk_known` | 对已知匹配 query 压低 unknownness        | 防止把 known 判成 unknown | 越低越好 | 如果这项异常，常见现象是 known→unknown 误分增多              |
| 伪未知 unknownness 损失     | `loss_unk_pseudo`              | 对 pseudo-unknown query 提升 unknownness | 强化 unknown 分支         | 越低越好 | 它在 pseudo_start 前为 0；之后如果始终几乎为 0，通常伪样本没挖到 |
| 可靠背景 unknownness 负损失 | `loss_unk_neg`                 | 对可靠背景压低 unknownness               | 防止背景被当 unknown      | 越低越好 | 太大说明背景抑制还不够稳                                     |
| 伪正物体性损失              | `loss_obj_pseudo`              | 对 pseudo-positive 提升 objectness       | 帮未匹配 query 学物体性   | 越低越好 | pseudo_start 后才应出现；若长期为 0，说明伪正太少            |
| 可靠背景物体性负损失        | `loss_obj_neg`                 | 对可靠背景压低 objectness                | 抑制背景误检              | 越低越好 | 如果它过大且 `K_AP50` 掉，可能压得太狠                       |
| 正交损失                    | `loss_orth`                    | 特征解耦正交约束                         | 服务第四章分支解耦        | 越低越好 | 看它下降时 `WI/AOSA` 是否同步改善                            |
| 去相关损失                  | `loss_decorr`                  | 预测层去相关约束                         | 降低 obj/unk/cls 相关性   | 越低越好 | 若这项下降但主指标不动，说明约束形式可能无效                 |

这些损失是否出现，取决于当前开了哪些开关；而 UOD 相关损失在 `pseudo_start` 和 `neg_start` 之前会被置 0。

# 伪监督与样本挖掘统计

| 指标名           | TensorBoard 路径/图名                                        | 含义                                        | 作用                                         | 好坏方向                       | 快速判断方法                                                 |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------- | -------------------------------------------- | ------------------------------ | ------------------------------------------------------------ |
| 伪正数量         | `train_stats/stat_num_dummy_pos` / `dummy_pos`               | 本轮被当成 pseudo-positive 的 query 数      | 看有没有给 unknown/objectness 提供正向弱监督 | 不是越大越好，**有且稳定**更好 | 长期接近 0 说明第三章主线基本没起作用；突然暴涨说明阈值可能太松 |
| 伪负数量         | `train_stats/stat_num_dummy_neg` / `dummy_neg`               | 本轮可靠背景负样本数量                      | 看背景抑制是否在工作                         | 有且适中更好                   | 若非常大，容易过度抑制；一直为 0，背景负监督没起作用         |
| 有效未匹配数     | `train_stats/stat_num_valid_unmatched` / `valid_unmatched`   | 经过基础过滤后可供挖掘的 unmatched query 数 | 看“可挖掘池”有多大                           | 适中更好                       | 它高而 `dummy_pos` 很低，说明筛选过严；它本身就很低，说明数据里可挖未知很少 |
| 正候选数         | `train_stats/stat_num_pos_candidates` / `pos_candidates`     | 满足伪正候选条件的数量                      | 看候选筛选强度                               | 适中更好                       | 长期 0 说明阈值或 known reject 过严                          |
| batch 选中伪正数 | `train_stats/stat_num_batch_selected_pos` / `batch_selected_pos` | batch 级动态分配后最终保留的伪正数          | 验证第三章“动态分配”有没有在发挥作用         | 适中更好                       | 应满足 `batch_selected_pos <= pos_candidates <= valid_unmatched` |
| 正阈值均值       | `train_stats/stat_pos_thresh_mean` / `pos_thresh_mean`       | 当前伪正阈值的平均值                        | 看阈值是否稳定                               | 稳定更好                       | 若忽高忽低，说明能量分布不稳定；若一直很高，可能选不出伪正   |

# 评估阶段的诊断图

| 文件名                                 | 含义                                          | 作用                                            | 快速判断方法                                                 |
| -------------------------------------- | --------------------------------------------- | ----------------------------------------------- | ------------------------------------------------------------ |
| `hist_probabilities.png`               | `obj_prob / unk_prob / cls_max` 三个直方图    | 看三条分支的分布形态                            | `obj_prob` 与 `cls_max` 不应完全同步，`unk_prob` 不应全塌到 0 |
| `scatter_relationships.png`            | `obj vs unk`、`obj vs cls_max` 散点图         | 看 objectness、unknownness、known score 的关系  | 若大量点同时高 `obj` 高 `cls_max` 高 `unk`，说明分支耦合严重 |
| `correlation_heatmap.png`              | `obj / unk / cls_max` 的相关性热图            | 验证第四章解耦思路                              | 第四章模型若有效，`unk` 与 `cls_max` 的相关性通常应下降      |
| `feature_pca.png` / `feature_tsne.png` | `proj_obj / proj_unk / proj_cls` 的特征投影图 | 看特征空间是否更可分                            | matched-known、unmatched-highunk、unmatched-other 三组若更分开，通常是好现象 |
| `current_metrics_bar.png`              | 当前 epoch 的评估柱状图                       | 快速看 U_R50、U_AP50、K_AP50、PK_AP50、WI、AOSA | 仅适合快速浏览，不适合严谨比较，因为量纲混在一起             |
| `qualitative_contact_sheet.png`        | 多张 overlay 定性图拼图                       | 论文和答辩展示                                  | 先看 unknown 是否被框出、是否大量错认成 known                |
